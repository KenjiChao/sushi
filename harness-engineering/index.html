駕馭工程：在以 Agent 為優先的世界中發揮 Codex 的力量

OpenAI · 2026 年 2 月 11 日 · 閱讀原文 →

在過去的五個月裡，我們的團隊一直在進行一項實驗：打造並發布一款軟體產品的內部 Beta 版，而其中沒有任何一行程式碼是手寫的。

這款產品有內部的每日活躍使用者，也有外部的 Alpha 測試人員。它同樣會發布、部署、發生故障，然後被修復。不同之處在於，每一行程式碼——包含應用程式邏輯、測試、CI 設定、文件、可觀測性（observability），以及內部工具——全都是由 Codex 撰寫的。我們估計，打造這個專案所花的時間，大約只有純手寫程式碼的十分之一。

人類負責引導方向，Agent 負責執行。

我們刻意選擇了這個限制，為的是逼迫自己打造出能讓工程開發速度呈現指數級增長所需的工具。我們只有幾週的時間來發布最終高達百萬行程式碼的專案。為了達成這個目標，我們必須去理解：當軟體工程團隊的主要工作不再是「寫程式」，而是「設計環境、給予明確意圖，並建立反饋迴圈讓 Codex Agent 能穩定工作」時，到底會發生什麼改變。

這篇文章記錄了我們與一群 Agent 團隊共同打造全新產品的學習歷程：我們發現了什麼會崩壞、什麼會產生複利效應，以及如何將我們唯一真正稀缺的資源——人類的時間與注意力——發揮到極致。

在 2025 年 8 月底，我們對一個空的 Repo 進行了第一次 commit。最初的專案骨架（scaffold）——包含 Repo 結構、CI 設定、程式碼格式化規則、套件管理器設定以及應用程式框架——都是在少數現有模板的引導下，由採用 GPT-5 的 Codex CLI 所生成的。甚至連一開始用來指導 Agent 如何在這個 Repo 中工作的 AGENTS.md 檔案，也是 Codex 自己寫的。

這個系統沒有任何預先由人類寫好的程式碼作為基礎。從一開始，這個 Repo 就是由 Agent 一手形塑的。

五個月後，這個 Repo 的程式碼規模已經達到百萬行級別，涵蓋了應用程式邏輯、基礎設施、工具、文件與內部開發者工具。在這段期間，我們僅靠著 3 名驅動 Codex 的工程師組成的小團隊，就開啟並合併了約 1,500 個 PR。換算下來，平均每位工程師每天能處理 3.5 個 PR，而且令人驚訝的是，隨著團隊現在成長到 7 名工程師，這個吞吐量還在持續增加。更重要的是，我們不是為了產出而產出：這款產品已經被內部數百名使用者所使用，其中還包含每天使用的內部重度使用者。

在整個開發過程中，人類從未直接貢獻過任何程式碼。這成為了團隊的核心理念：零手寫程式碼。

不讓人類親自動手寫程式，反而帶來了一種截然不同的工程工作：專注於系統、鷹架與槓桿效應。

早期的進展比我們預期的還要慢，這不是因為 Codex 能力不足，而是因為環境的定義不夠明確。Agent 缺乏邁向高階目標所需的工具、抽象化概念與內部結構。我們工程團隊的主要工作，變成了「如何賦能 Agent 去做有用的工作」。

在實務上，這意味著要以深度優先（depth-first）的方式工作：將大型目標拆解成較小的積木（設計、程式碼、審查、測試等），透過提示（prompt）讓 Agent 構建這些積木，再用它們來解鎖更複雜的任務。當出現錯誤時，解決方案幾乎從來不是「再努力試試看」。因為取得進展的唯一方法是讓 Codex 完成工作，人類工程師總是會介入並思考：「這裡缺少了什麼能力？我們該如何讓這個能力對 Agent 來說既容易理解，又能被強制執行？」

人類幾乎完全透過提示來與系統互動：工程師描述任務、執行 Agent，並讓它開啟一個 PR。為了將 PR 推進到完成，我們會指示 Codex 在本地端審查自己的變更，並在本地端與雲端要求特定的其他 Agent 進行額外審查，回應任何來自人類或 Agent 的反饋，然後在這個迴圈中持續迭代，直到所有 Agent 審查者都滿意為止。Codex 會直接使用我們的標準開發工具（如 gh、本機腳本以及內建於 Repo 的技能）來收集上下文，不需要人類複製貼上到 CLI 裡。

人類可以審查 PR，但這不是必須的。隨著時間推移，我們已經將幾乎所有的審查工作推向了 Agent 對 Agent 的處理模式。

隨著程式碼吞吐量的增加，我們的瓶頸變成了人類的 QA（品質保證）能力。既然人類的時間與注意力是固定的限制條件，我們就努力為 Agent 添加更多能力，讓應用程式的 UI、日誌（logs）與指標（metrics）本身，都能直接被 Codex 所讀懂（legible）。

例如，我們讓應用程式能夠針對每個 git worktree 獨立啟動，這樣 Codex 就可以為每一個變更啟動並驅動一個獨立的實例。我們還將 Chrome DevTools Protocol 串接到 Agent 執行環境中，並建立處理 DOM 快照、螢幕截圖與導航的技能。這讓 Codex 能夠直接重現 Bug、驗證修復結果，並對 UI 的行為進行推理。

我們也對可觀測性工具做了同樣的事。我們透過針對每個 worktree 都是臨時（ephemeral）的本地可觀測性技術堆疊，將日誌、指標與追蹤數據暴露給 Codex。Codex 在一個完全隔離的應用程式版本上運作——包含它專屬的日誌與指標，這些都會在任務完成後被銷毀。Agent 可以使用 LogQL 查詢日誌，並使用 PromQL 查詢指標。有了這些上下文，像「確保服務在 800 毫秒內完成啟動」或「確保這四個關鍵使用者旅程中的任何一段追蹤（span）不超過兩秒」這類的提示，就變得可以被處理了。

我們經常看到單次的 Codex 執行會在一個任務上連續工作超過六個小時（通常是在人類睡覺的時候）。

在讓 Agent 能有效處理大型且複雜任務的過程中，上下文管理是最大的挑戰之一。我們學到的第一個教訓很簡單：給 Codex 一張地圖，而不是一本 1,000 頁的說明書。

我們嘗試過「一個巨大的 AGENTS.md」這種做法。它以可預期的方式失敗了：

  • 上下文是一種稀缺資源。一個巨大的指令檔案會排擠任務、程式碼以及相關文件的空間——導致 Agent 要麼錯過關鍵限制，要麼開始對錯誤的目標進行最佳化。
  • 過多的指導等同於沒有指導。當每件事都「很重要」時，就沒有什麼是重要的。Agent 最終會變成只在局部進行模式匹配，而不是有目的性地導航。
  • 它會瞬間腐敗。一個單體式的巨大手冊會變成過期規則的墳墓。Agent 無法分辨哪些內容仍然正確，人類也停止維護它，這個檔案悄悄地就變成了一個誘人的麻煩。
  • 很難驗證。單一的一大包文字不利於進行機械式檢查（覆蓋率、新鮮度、所有權、交叉連結），因此內容偏離（drift）是不可避免的。

因此，與其把 AGENTS.md 當作百科全書，我們改將它視為目錄。

Repo 的知識庫存放在一個結構化的 docs/ 目錄中，並被視為事實來源（system of record）。我們將一份簡短的 AGENTS.md（大約 100 行）注入到上下文中，它主要作為一張地圖，指向其他地方更深層的事實來源。

設計文件會被分類和建立索引，包含驗證狀態以及一套定義 Agent 優先操作原則的核心理念。架構文件則提供了業務領域和套件分層的頂層地圖。品質文件會對每個產品領域和架構層進行評分，並追蹤隨時間推移產生的落差。

計畫（Plans）被視為一等公民的產物。針對小型變更，我們會使用臨時的輕量級計畫；而複雜的工作則會被記錄在執行計畫中，這些計畫包含了進度和決策日誌，並會被 commit 到 Repo 裡。活躍的計畫、已完成的計畫以及已知的技術債都會被版本化並放在一起，讓 Agent 可以在不依賴外部上下文的情況下運作。

這實現了漸進式揭露（progressive disclosure）：Agent 從一個小而穩定的切入點開始，並被教導接下來該去哪裡尋找資訊，而不是一開始就被過多的資訊淹沒。

我們透過機械式手段來強制執行這一點。專屬的 linters 和 CI 工作會驗證知識庫是否保持在最新狀態、是否有正確的交叉連結，並且結構無誤。一個定期執行的「文件園丁（doc-gardening）」Agent 會掃描無法反映真實程式碼行為的過期或廢棄文件，並開啟 PR 來進行修復。

隨著程式碼庫的演進，Codex 的設計決策框架也需要跟著進化。因為這個 Repo 完全由 Agent 生成，所以它首先是為了 Codex 的易讀性而最佳化的。就像團隊會為了讓新進工程師更容易導覽程式碼而改善結構一樣，我們人類工程師的目標，是讓 Agent 能夠直接從 Repo 本身去推演完整的業務領域知識。

從 Agent 的角度來看，任何它在執行時無法在上下文中存取的東西，等同於不存在。存在於 Google Docs、聊天紀錄或人類腦袋裡的知識，系統是無法存取的。存在於 Repo 本地端、有版本控制的產物（例如程式碼、markdown、資料綱要、可執行的計畫），才是它唯一能看見的。

我們學到，隨著時間的推移，我們需要將越來越多的上下文推播到 Repo 中。那段讓團隊對某個架構模式達成共識的 Slack 討論呢？如果它不能被 Agent 發現，那它就無法被讀懂，就像三個月後才加入的新進員工一樣會對此一無所知。

給 Codex 更多上下文，意味著要組織並揭露正確的資訊，讓 Agent 能夠對此進行推理，而不是用臨時的指令淹沒它。就像你會向新團隊成員介紹產品原則、工程規範和團隊文化一樣，將這些資訊提供給 Agent，會帶來更符合期望的產出。

這個思考框架釐清了許多權衡。我們偏好那些能夠完全內化，並且可以在 Repo 內進行推理的相依性套件和抽象化。那些常被形容為「無聊」的技術，通常因為具備良好的可組合性、API 穩定性以及在訓練資料集中的高代表性，反而更容易被 Agent 建模。在某些情況下，讓 Agent 重新實作部分功能，會比繞過公共函式庫那些不透明的上游行為來得更划算。

將更多系統內容轉換為 Agent 可以直接檢查、驗證和修改的形式，能大幅提升槓桿效應——不只是為了 Codex，也是為了在程式碼庫上工作的其他 Agent。

光靠文件並不能保持一個完全由 Agent 生成的程式碼庫的一致性。藉由強制執行不變性（invariants），而不是微觀管理實作細節，我們讓 Agent 能夠快速發布功能，又不會破壞基礎。

Agent 在擁有嚴格邊界和可預測結構的環境中最有效率，因此我們圍繞著一個嚴謹的架構模型來構建應用程式。每個業務領域都被劃分為一組固定的層級，具有嚴格驗證的相依方向以及一組有限的允許連結。我們透過自訂的 linters 和結構測試，以機械式的方式來強制執行這些限制。

這通常是那種你會推遲到擁有數百名工程師時才會去做的架構。但對於寫程式的 Agent 來說，這是早期的先決條件：這些限制正是確保速度之餘，又不會造成程式碼腐壞或架構偏離的原因。

在實務上，我們透過自訂 linters、結構測試以及一小部分「品味不變性（taste invariants）」來強制執行這些規則。舉例來說，我們透過自訂 lints 以靜態方式強制要求結構化日誌、綱要與型別的命名慣例、檔案大小限制，以及特定平台的可靠性要求。因為這些 lints 是自訂的，我們在撰寫錯誤訊息時，就會把修復指令注入到 Agent 的上下文中。

在以人類為優先的工作流程中，這些規則可能會讓人覺得過於學究或充滿限制。但在 Agent 身上，它們變成了乘數：一旦編寫完成，就能同時套用到所有地方。

同時，我們也非常明確地區分出哪些地方需要限制，哪些地方不需要。這就像在領導一個大型的平台工程組織：在中央強制執行邊界，在局部允許自治。

最終產生出來的程式碼並不總是符合人類的風格偏好，但這沒關係。只要產出是正確的、可維護的，並且對未來的 Agent 執行來說是容易讀懂的，這就符合標準了。

人類的品味會持續反饋到系統中。審查評論、重構 PR 以及使用者遇到的 Bug，都會被捕捉並轉化為文件更新，或是直接編碼進工具中。當文件不足以規範時，我們就會將規則提升到程式碼的層級。

隨著 Codex 的吞吐量增加，許多傳統的工程規範反而變得適得其反。

這個 Repo 以最少的阻擋合併關卡（blocking merge gates）來運作。PR 的生命週期都很短。不穩定的測試（Test flakes）通常會透過後續的執行來解決，而不是無限期地阻擋進度。在一個 Agent 吞吐量遠超過人類注意力的系統中，修正是很廉價的，而等待則是非常昂貴的。

在一個低吞吐量的環境中，這種做法可能是不負責任的。但在這裡，這通常是正確的權衡。

當我們說這個程式碼庫是由 Codex Agent 生成的，我們指的是程式碼庫裡的所有東西。Agent 會產出：

  • 產品程式碼與測試
  • CI 設定與發布工具
  • 內部開發者工具
  • 文件與設計歷史
  • 評估測試框架（Evaluation harnesses）
  • 審查評論與回應
  • 管理 Repo 本身的腳本
  • 正式環境儀表板定義檔

人類始終保持在迴圈中，但我們工作的抽象層次與以往不同了。我們負責排定工作優先順序、將使用者反饋轉譯為驗收條件，並驗證結果。當 Agent 遇到困難時，我們會將其視為一個訊號：找出缺少了什麼——是工具、護欄（guardrails），還是文件——然後將其反饋回 Repo 中，而且永遠都是讓 Codex 自己來寫這些修復。

隨著越來越多的開發迴圈（測試、驗證、審查、反饋處理與復原）被直接編碼進系統中，這個 Repo 最近跨過了一個極具意義的門檻：Codex 已經能夠端到端地推動一個新功能。

現在，只要給定一個單一提示，Agent 就能夠：

  • 驗證程式碼庫的當前狀態
  • 重現回報的 Bug
  • 錄製示範錯誤的影片
  • 實作修復方案
  • 透過驅動應用程式來驗證修復結果
  • 錄製第二支示範解決結果的影片
  • 開啟 PR
  • 回應 Agent 與人類的反饋
  • 偵測並修復建置失敗
  • 只有在需要判斷時才升級交給人類
  • 合併變更

這種能力高度依賴這個 Repo 特定的結構與工具，因此不該假設在沒有類似投資的情況下也能通用——至少目前還不行。

完全的 Agent 自治也帶來了新的問題。Codex 會複製 Repo 中已存在的模式——即便是那些參差不齊或非最佳的模式。隨著時間的推移，這無可避免地會導致偏離。

一開始，人類會手動解決這個問題。我們的團隊過去常常花費每個星期五（一週 20% 的時間）來清理「AI 產生的垃圾碼（AI slop）」。毫不意外地，這種做法無法擴展。

取而代之的是，我們開始將我們所謂的「黃金原則」直接編碼進 Repo 中，並建立了一個定期執行的清理流程。這些原則帶有主觀意見，且都是機械式的規則，能確保程式碼庫對未來的 Agent 執行保持易讀與一致。舉例來說：（1）我們偏好共用的公用套件（utility packages）而不是手寫的輔助函式，以便集中管理不變性；（2）我們不會以「YOLO 風格」來隨意探測資料——我們會驗證邊界或依賴有型別的 SDK，這樣 Agent 就不會意外地把功能建立在對結構的猜測上。我們有一組背景 Codex 任務會定期掃描是否有偏離規則的情況、更新品質評分，並開啟針對性的重構 PR。這其中大部分的 PR 都能在不到一分鐘內完成審查並自動合併。

這就像是垃圾回收（garbage collection）的機制。技術債就像高利貸：持續以小幅度償還，幾乎總是比讓它產生複利、最後再一次痛苦地解決要好得多。人類的品味被捕捉一次後，就會持續強制套用在每一行程式碼上。這也讓我們能夠每天捕捉並解決不良的模式，而不是讓它們在程式碼庫中蔓延好幾天甚至好幾週。

到目前為止，這項策略在 OpenAI 的內部發布與採用過程中都運作得很好。為真實使用者打造真實產品，有助於將我們的投資扎根於現實，並引導我們邁向長期的可維護性。

我們還不知道的是，在一個完全由 Agent 生成的系統中，架構的連貫性在幾年後會如何演變。我們仍在學習人類的判斷在哪裡能發揮最大的槓桿作用，以及如何將這種判斷編碼起來讓它產生複利。我們也不知道隨著模型能力與日俱增，這個系統未來會如何發展。

有一點是明確的：開發軟體依然需要紀律，但這種紀律更多是展現在鷹架上，而不是程式碼裡。那些能讓程式碼庫保持連貫的工具、抽象化與反饋迴圈，正變得越來越重要。

我們現在最困難的挑戰，集中在設計環境、反饋迴圈與控制系統上，這些都是為了幫助 Agent 達成我們的目標：大規模地開發與維護複雜且可靠的軟體。

隨著像 Codex 這樣的 Agent 承擔越來越多的軟體生命週期，這些問題將變得更加重要。我們希望分享這些早期的學習經驗，能幫助你思考該把精力投資在哪裡，好讓你能夠專心打造事物。
