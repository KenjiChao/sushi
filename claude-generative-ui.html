逆向工程 Claude 的生成式 UI - 並為終端機打造專屬版本

michaellivs.com · 翻譯：翻譯官 📝 · 2026-03-13

發現 (The Discovery)

Anthropic 在幾個小時前發表了 Claude 的生成式 UI (generative UI)。互動式小工具 (interactive widgets)——滑桿、圖表、動畫——在 claude.ai 的對話中行內渲染 (rendered inline)。不是圖片，也不是程式碼區塊 (code blocks)。而是能在聊天室內執行 JavaScript 的真實 HTML 應用程式。

這並不令人意外。Vercel 等公司推動生成式 UI 已經有一段時間了，而且我知道 Anthropic 遲早會在這個領域有所動作。這也不是我第一次深入研究 Anthropic 的實作細節——我之前就曾對他們的沙盒架構 (sandbox architecture) 進行過逆向工程 (reverse-engineered) 並撰寫了相關探討沙盒的文章。

所以我帶著一個明確的目的前往 claude.ai：確切了解他們是如何實作的。最終，我為終端機為基礎的程式碼代理 (terminal-based coding agent) pi 打造了屬於我自己的版本。

第一部分：盤問 Claude 關於它自己的 UI

是工具調用，而不是 Markdown

我的第一個假設錯了。我原以為 Claude 是將 HTML 作為其 Markdown 回應的一部分輸出，然後由前端 (frontend) 進行行內渲染。Claude 糾正了我：「它根本不是 Markdown 輸出的一部分。我調用了一個名為 show_widget 的工具 (tool call)，並將 HTML 作為參數傳遞。」

所以這是一個工具調用。HTML 是一個參數負載 (parameter payload)，而不是串流文本 (streamed text)。包含四個參數：

  • i_have_seen_read_me：Claude 必須先調用 read_me 來載入設計指南 (design guidelines)。
  • title：蛇形命名法 (snake_case) 的識別符 (identifier)。
  • loading_messages：1 到 4 個短字串，會在渲染時顯示。
  • widget_code：原始 HTML 片段 (raw HTML fragment)，不包含 html/head/body/doctype 標籤。

read_me 模式 - 漸進式揭露 (Progressive Disclosure)

在使用 show_widget 之前，Claude 必須帶有 modules 參數來調用 read_me。可用的模組有：diagram、mockup、interactive、chart、art。這是一個延遲載入的文件系統 (lazy documentation system)——只在需要時載入相關的子集，而不是一開始就傾倒整個設計系統。

不是 Iframe - 即時 DOM 注入 (Live DOM Injection)

證明它是 DOM 注入而不是 iframe 的證據有：

  • CSS 變數可以正常解析（處於同一個文件層疊 cascade）。
  • 可以存取 sendPrompt()（父頁面的函式）。
  • 透明背景（沒有 iframe 容器）。
  • 沒有載入時的閃爍（沒有 iframe 邊框或捲軸）。

所謂的「沙盒」其實只是一個內容安全策略 (CSP, Content Security Policy)，用來限制可用的內容傳遞網路 (CDN, Content Delivery Network) 網域：cdnjs.cloudflare.com、cdn.jsdelivr.net、unpkg.com、esm.sh。

Artifacts 與 Visualizer 的比較

Artifacts = 交付成果（側邊面板、固定的函式庫、具持久性 persists）。Visualizer = 行內增強功能（透明、可使用任何 CDN 函式庫、具短暫性 ephemeral）。兩者的觸發語句也不同。

串流架構 (Streaming Architecture)

  1. 大型語言模型 (LLM) 生成 show_widget 工具調用。
  2. widget_code 以 token 為單位進行串流 (streaming)。
  3. 對部分內容進行漸進式 HTML 解析 (incremental HTML parsing)。
  4. 即時插入 DOM 節點。
  5. CSS 變數立即解析。
  6. 樣式/HTML 在到達時隨即渲染，腳本 (scripts) 最後執行。
  7. CDN 函式庫非同步 (async) 載入。

第二部分：為 Pi 打造專屬版本

Pi 是一個以終端機為基礎的程式碼代理，無法直接渲染互動式 HTML。被否決的選項：終端機圖片協定（無互動性）、本機網頁伺服器（會跳出終端機）、TUI（終端機使用者介面）近似方案（限制太多）。

解決方案：Glimpse——原生的 macOS 微型 UI 函式庫 (micro-UI library)。透過 Swift 執行檔加上 Node.js 包裝器 (wrapper)，能在 50 毫秒內開啟 WKWebView 視窗。不使用 Electron。具備完整的瀏覽器引擎、雙向 JSON、setHTML() 以及 send(js) API。

仿效 Claude 的兩個工具：visualize_read_me（延遲文件載入器）與 show_widget（開啟 Glimpse 視窗，回傳互動資料）。

第三部分：串流挑戰

4 次嘗試：

  1. 每個變化片段 (delta) 都調用 setHTML() → 卡頓，每 80 毫秒就會發生全頁面重排 (full page reflow)。
  2. 透過 JavaScript eval 修改 innerHTML → 較好，但依然卡頓。
  3. 樸素的 DOM 附加 (Naive DOM appending) → 失敗，因為瀏覽器會自動補齊未閉合的標籤。
  4. 使用 morphdom 進行 DOM 差異比對 (DOM diffing) → 完美解決方案。

morphdom 應用了最小化的修補 (minimal patches)：跳過未變更的節點（isEqualNode），對新節點套用淡入動畫。透過待處理緩衝區 (pending buffer) 解決了競爭危害 (race condition)。腳本執行方面：在工具調用結束 (toolcall_end) 時，將 script 標籤複製到全新的元素中。

第四部分：萃取 Anthropic 的設計指南

在瀏覽器開發者工具 (devtools) 中找到了完整的 read_me 逐字結果。模組大小：interactive=19K, chart=22K, mockup=19K, art=17K, diagram=59K。

關鍵規則：串流優先（樣式 → HTML → 腳本最後）、禁止使用漸層/陰影/模糊 (gradients/shadows/blur)、兩種字體粗細 (400/500)、強制暗色模式 (dark mode mandatory)、9 種色階 × 7 個停靠點 (color ramps × 7 stops)、嚴格的 SVG 規則、針對圖表的複雜度預算 (complexity budgets)。

第五部分：經驗總結 (Lessons)

  1. 生成式 UI 比看起來簡單——只不過是工具調用 + DOM 注入。
  2. read_me 模式非常聰明——按需將文件延遲載入到上下文 (context) 中。
  3. DOM 差異比對解決了串流的流暢度問題。
  4. Glimpse 搭起了終端機與瀏覽器之間的橋樑。
  5. 正規化的串流事件 (normalized streaming events) 適用於所有 LLM 供應商。

程式碼：約 350 行 TypeScript。安裝方式：

pi install npm:pi-generative-ui
